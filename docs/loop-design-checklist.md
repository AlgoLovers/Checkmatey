# 루프 설계 체크리스트

새 루프를 만들거나 기존 루프를 손보기 전에 이 8개 질문을 모두 통과시킨다.
(참고: awesome-loop-engineering, cobusgreyling/loop-engineering)

1. **목표 하나를 말한다** — "개선"이 아니라 구체적 결과 하나.
2. **인테이크를 정의한다** — 일이 어디서 들어오나(PR, CI 실패, 이슈, 로그, 스케줄).
3. **실행을 격리한다** — worktree / 브랜치 / 읽기 전용 중 하나.
4. **검증 신호를 먼저 쓴다** — 재시도 전에 테스트·린트·빌드가 존재.
5. **상태를 모델 밖에 저장한다** — `STATE.md`, 커밋, 이슈 코멘트가 컨텍스트 윈도우를 넘어 남는다.
6. **maker와 checker를 분리한다** — 실행한 에이전트가 아닌 다른 것이 "됐음"을 검증.
7. **자율성에 예산을 건다** — 턴/재시도/토큰/시간 상한 (loop-constraints.md).
8. **에스컬레이션을 설계한다** — 언제 PR 열고, 이슈 파고, 사람에게 묻고, 멈추는가.

## 루프 성숙도 모델 (0→6)

- L0 수동 프롬프트 → L1 스크립트 재시도 → L2 스케줄 → L3 상태 저장 →
  L4 자기검증 → L5 멀티에이전트 → L6 프로덕션 감독(관측·예산·승인·롤백)

Checkmatey는 지금 **L1**(보고 전용). 검증 게이트가 신뢰되면 개발 루프(LOOP-1)를 L2로 올린다.

## 흔한 실패 모드

- 검증 신호 없이 재시도 → 무한 루프. **신호 먼저.**
- 상태를 컨텍스트에만 → 세션 끝나면 소실. **파일에 저장.**
- 조용한 상한(top-N/샘플링)을 로그 없이 → "다 했다"로 오독. **드롭한 건 로그로 남긴다.**
- 같은 루프, 다른 운영자, 반대 결과 → 루프는 맥락을 모른다. `STATE.md`로 맥락을 공유.
